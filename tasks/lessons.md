[2026-06-25] | WC lineups for live matches weren't on Neon (cloud app couldn't show them). The dispatcher DID capture them (matches 23/24, 15:14, 22 starters each) but wrote them to LOCAL SQLite — a split-brain. Root cause: run_wc_dispatcher.sh loads creds via `set -a; source .env`, and the Neon DATABASE_URL ends in an unquoted `&` (`...?sslmode=require&channel_binding=require`). Bash treats `&` as a background operator, so `DATABASE_URL=...require` runs as a backgrounded assignment in a throwaway subshell → DATABASE_URL ends up UNSET in the parent → db.py's Priority-3 fallback uses local SQLite (it logged "No DATABASE_URL — falling back to LOCAL ... split-brain risk"). The WC pipeline was unaffected because pipeline.py calls load_dotenv() itself (python-dotenv parses `&` fine); the dispatcher was the ONE entrypoint relying only on the shell source. | NEVER load a .env containing URLs into a shell via `source`/`set -a` — an unquoted `&`/`?`/`#`/space silently mangles or drops the value (here it backgrounded the assignment → UNSET). Two fixes: (1) every Python entrypoint should call `load_dotenv(Path(__file__).resolve().parents[N] / ".env")` itself (python-dotenv is quote/`&`-safe) — match the existing pipeline.py pattern; (2) quote values in .env (`DATABASE_URL="...&..."`). Detect split-brain by comparing a table's row coverage on Neon vs local (`SELECT match_id, COUNT(*) ... GROUP BY` on both); if local has rows Neon doesn't for a live table, a pipeline wrote to the wrong DB. When db.py logs the split-brain WARNING, treat it as a real incident.

[2026-06-25] | Backtest ROI under COMPOUNDING flat staking (stake = running bankroll × %) is BIASED: when a strategy loses, the bankroll death-spirals toward $0 and stakes shrink with it, so losses stop accumulating and the apparent ROI is FLATTERED. Found when aligning the backtester to level staking — EPL backtest read −12.7% compounding vs −17.8% level; Ligue1 −27.4% vs −37.0%; deltas up to −9.5pp. | For backtest ROI use LEVEL staking (constant stake off the STARTING bankroll) — it's the honest per-unit edge and isn't path-dependent. Compounding makes ROI depend on the ORDER of wins/losses and dampens losses (flatters losers). ALWAYS sanity-check a backtest's final bankroll path: if it crashed to ~$0, the compounding ROI is meaningless. (Brier is staking-independent — unaffected.) Corollary: when measuring a change's effect, isolate it — here, replaying ONE backtest's realised bet sequence under both staking methods (same bets/results, only stake sizes differ) gave the clean delta without re-running or re-training.

[2026-06-26] | (2nd occurrence of the .env-sourcing class — see 2026-06-25 `&` lesson.)
