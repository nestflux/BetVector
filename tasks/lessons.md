[2026-06-25] | WC standings diagnosis: I concluded "the WC pipeline hasn't run since 23-Jun" because max(WCMatch.updated_at) was the bulk-load time — WRONG. WCMatch.updated_at has server_default=func.now() but NO onupdate=func.now(), so it only records INSERT time and never reflects UPDATEs; the pipeline log actually showed "WC results: 36 updated" on 24-Jun. The real cause was the once-daily morning cadence (4 matches finished after the 09:30 scrape). | Never use a model's updated_at as a "last modified / pipeline ran" signal unless the column has onupdate=. [2026-06-25] | RBAC/multi-user: onboarding AND Settings let ANY user toggle GLOBAL state — League.is_active (which drives the shared data pipeline) and injury flags (which skew everyone's predictions) — because the app was built single-user. [2026-06-24] | EMAIL-OPTIN: to flip a NEW-user default (notify_morning/evening/weekly on→off) on an ALREADY-DEPLOYED DB, changing the model's `server_default` alone is NOT enough — server_default only applies at create_all (fresh schemas); the existing column keeps its old DB-level DEFAULT (1), so new ORM inserts would still get 1. | Add the Python-side `default=0` to the Column (alongside server_default): SQLAlchemy applies `default=` on every ORM insert and overrides the DB default, so new users come out off on the live DB with NO ALTER COLUMN (which SQLite can't do anyway). Verified by creating a probe user (rolled back) against the existing-schema local DB. Also remember UI creation paths (onboarding wizard toggles + their session_state fallbacks) carry their own defaults — flip those too, or the wizard re-opts the user in.

[2026-06-24] | WC-EMAIL: adding a NOT-NULL column (notify_wc) to the User model broke 11 e34 auth tests with "no such column: users.notify_wc" — because admin.py queries users at IMPORT time, and that hit the local data/betvector.db backup, whose schema predated the column (the migration only auto-applies when init_db runs). | When adding a column to any model, apply the migration to BOTH the local SQLite dev DB AND Neon immediately (run init_db with all model modules imported, once per DB as separate processes — the engine singleton caches the first URL). Import-time DB queries in views mean a stale local backup will fail the suite even though the ORM/migration are correct. The model + migration entry alone isn't enough; the live schemas must be migrated too.

[2026-06-24] | DH-01 data-health engine: the first version flagged league sources (predictions/results/stats/weather) as FAIL because they were weeks stale — but it was the June off-season, so that staleness was expected, not broken. The live smoke against Neon showed 4 spurious FAILs that would have made the owner distrust the tool. | A data-FRESHNESS check must be ACTIVITY-aware: gate "source is stale" on whether the source SHOULD be flowing right now (e.g. leagues in-season — any upcoming fixture or a match finished in the last few days), and SKIP rather than FAIL when it's legitimately idle. And ALWAYS smoke a new monitoring check against real production data before trusting its verdicts — false positives are how a health tool loses the owner's trust.
